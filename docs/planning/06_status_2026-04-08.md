# Implementation Status — 2026-04-08

## What's Been Built

### Core MCP Server + CLI
- Job orchestration with DAG-based execution, heartbeat stall detection, configurable concurrency
- SQLite persistence (WAL mode) for jobs, events, steer messages, config
- 16 MCP tools (see CLAUDE.md for key tools)
- Three agent adapters: ClaudeAdapter (`claude --print`), OpenCodeAdapter, CodexAdapter
- CLI with `init`, `serve`, `status`, `config` commands

### Bug Fixes Applied (from 05_test_observations)
Commit `3dbd88b`:
1. **Auto-sync status files** — `status.md` and `job-queue.md` update on every job state transition, server-side
2. **`tenet status` reads live DB** — SQLite as primary source, markdown as fallback
3. **`tenet_retry_job` tool** — resets completed/failed jobs to pending, preserves DAG linkage, supports enhanced prompt for retries
4. **Dev job prompt preamble** — `dev` type jobs get explicit "Deliverable Requirements" instructions so workers implement instead of explore

### Multi-Session Document Structure
Commit `0e5642c`:
- Documents use `$date-$feature.md` naming (e.g. `spec/2026-04-08-oauth.md`)
- `tenet_compile_context` resolves latest doc per feature via glob + date sort
- Falls back to old singleton format (`spec/spec.md`) for migration
- `tenet_register_jobs` requires a `feature` slug that propagates to all jobs
- Decomposition moved to its own directory (`decomposition/`) separate from `spec/`
- No index.md — filesystem is the index
- See `docs/planning/multi-session-document-structure.html` for full visual explanation

### Server-Side Loop Driver (Explicitly Deferred)
The orchestrator agent is still the bottleneck. When it stalls (context compaction after ~2 hours), the pipeline stops. The proposed fix is moving the orchestration loop into the MCP server, but this is a major architecture change. See `05_test_observations_2026-04-08.md` "Server-Side Loop Driver — Design Discussion Points" for the open questions.
