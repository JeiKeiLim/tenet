## Issue 1: Adapter timeout too short (10 min cap)

**Evidence**: 3 playwright_eval jobs failed at exactly `10m 0s` with "claude invocation timed out after 600000ms". Status showed: `full-e2e-final`, `playwright_eval for profile-settings-and-public-profile`, `playwright_eval for share-card-modal-downloads`, `playwright_eval for badge-grid-and-celebration`.

**Root cause**: The Claude adapter (and likely all adapters) has a hardcoded 600000ms timeout for subprocess invocation. Playwright e2e jobs that need to start a dev server, run scripted tests, AND do exploratory testing easily exceed 10 minutes.

**Fix options**:
- A) Make timeout configurable per job type (eval jobs get longer timeout)
- B) Increase default timeout globally (e.g., 30 min)
- C) Both — higher default + per-type override

**Status**: [x] Done — default raised to 30 min, overridable via `tenet config --timeout <minutes>`

## Issue 2: Stall detection killing eval jobs

**Evidence**: 3 eval jobs for `badge-system` (code_critic, test_critic, playwright_eval) all failed with "stall detected". These are independent eval workers that may run for extended periods without calling MCP tools (they're reading code, analyzing, writing findings).

**Root cause**: Stall detection uses heartbeat — if a job doesn't call an MCP tool within the stall window, it's considered stalled. Eval workers that are doing pure analysis (reading files, thinking) won't send heartbeats during that time.

**Fix options**:
- A) Increase stall timeout for eval job types
- B) Have the adapter send periodic heartbeats automatically (not dependent on MCP tool calls)
- C) Disable stall detection for eval jobs entirely
- D) Agents send explicit "still working" heartbeat via MCP tool

**Status**: [x] Done — server_id approach: each server instance gets a UUID, orphaned jobs (different server_id) reset to pending on startup. Heartbeat timeout bumped to 30 min.

## Issue 4: "Not inside a trusted directory" git error

**Evidence**: One eval job (`de6e7028`) failed with "Not inside a trusted directory and --skip-git-repo-check was not specified."

**Root cause**: When the adapter spawns a subprocess in the project directory, git's `safe.directory` check rejects it. This is a git security feature for directories owned by different users.

**Fix options**:
- A) Pass `--skip-git-repo-check` flag when spawning adapters
- B) Set `GIT_CONFIG_GLOBAL` or inject safe.directory in subprocess env
- C) Document as a setup requirement (user must add directory to git safe.directory)
- D) Investigate — this only happened once out of 187 jobs, so it may be environment-specific

**Status**: [x] Documented — one-off issue, covered in tenet:diagnose skill. Agent can self-fix by running `git config --global --add safe.directory <path>`.
