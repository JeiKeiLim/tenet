Agile Mode — Design Sketch

tenet · alignment draft · 2026-04-27

Optional second mode for Tenet's autonomous loop. The fully-autonomous loop stays the default. Agile mode pauses after each usable slice so the user can run the app, give feedback, and steer the next slice. Reuses the existing 7-phase pipeline — only decomposition output shape and a checkpoint mechanism are new.

1 · Same pipeline, two DAG shapes

Decomposition reads a mode flag from the spec and produces a different DAG. Everything before decomposition is identical; everything inside each job (eval, retries, etc.) is identical.

2 · Single spec doc, slice-structured inside

No new artifact lane. Same spec/{date}-{feature}.md file, just with a slice plan section. The plan evolves as feedback comes in — past slices stay frozen, future slices get rewritten.

3 · Agile loop — reuse pipeline, pause after each slice

After each slice's per-job eval passes, the loop pauses. The user runs the app. The user's response routes back into the existing pipeline at whichever phase fits the feedback.

What's actually new vs reused

NEW (4 small additions)

  • Mode flag on the spec — autonomous | agile. Picked after interview, stored in the spec doc.
  • Decomposition reads the mode — produces a sliced DAG (sequence of usable products) when agile, today's component DAG when autonomous.
  • Pause/resume mechanism — checkpoint event after each slice's eval passes; agent waits for user input.
  • Spec template gets a slice-plan section — bigger picture present from the start, evolves with feedback.

REUSED (unchanged)

  • All 7 phases (interview · spec · mockup · harness · decomposition · execution · eval)
  • Single spec document per feature
  • Per-job eval (critic + test critic + playwright_eval)
  • All MCP tools — including tenet_register_jobs, tenet_compile_context, tenet_add_steer
  • SQLite state, status writers, retry logic, orphan recovery

Open questions still on the table

Q1 · First-slice bundling rule
A slice must be actually usable. Login alone isn't usable without signup + sessions. The planner should be allowed to bundle dependent sub-features when they're needed for the main capability to work. Worth stating as an explicit rule in the agile-mode decomposition prompt.
Q2 · Re-entry phase selection
When the user redirects, who picks which phase to re-enter — interview, spec, or just decomposition? Cleanest: agent proposes based on the redirect content, user can override.
Q3 · End condition
Agile mode is open-ended (user keeps asking for next slice). Done = user explicitly chooses "done" at a checkpoint. Worth deciding whether the original slice plan acts as a soft target or just as direction.
