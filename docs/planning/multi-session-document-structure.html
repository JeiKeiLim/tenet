What Changes Scope

Directory Scope Naming Who Writes Notes
spec/ per-feature $date-$feature.md Agent (crystallization phase) One file per feature per run
decomposition/ per-feature $date-$feature.md Agent (decomposition phase) Moves out of spec/ to its own dir
interview/ per-feature $date-$feature.md Agent (interview phase) Transcript + decisions
knowledge/ per-finding $date_$slug.md MCP server (tenet_update_knowledge) Already works this way. No change.
harness/ project-wide current.md User (manual) Engineering constraints. Singular.
steer/ project-wide inbox.md User + Agent Messages are project-scoped. Singular.
status/ auto-generated status.md, job-queue.md MCP server (on state change) Reflects current DB state. Always overwritten.

How tenet_compile_context Resolves Documents

When compiling context for a job, the MCP server needs to find the right spec, decomposition, and knowledge. Here's the resolution logic:

1
Job has feature: "oauth" in its params (set during tenet_register_jobs)
2
Glob spec/*-oauth.md → finds 2026-04-08-oauth.md, 2026-05-01-oauth.md
3
Sort by date prefix, pick latest → spec/2026-05-01-oauth.md
4
Same for decomposition: decomposition/2026-05-01-oauth.md
5
Knowledge: glob knowledge/*oauth* or knowledge/*token* — agent decides which are relevant by filename. Not all are loaded.
6
Harness: always harness/current.md (project-wide)
No index.md, no DB initiative table, no "active feature" state. The job knows its feature name → glob resolves the documents → date sorts pick the latest.

Edge Cases

What if two features run on the same day?
The feature name differentiates: 2026-04-08-oauth.md vs 2026-04-08-payments.md. Date is for chronological sorting, feature name is for identification.
What if a feature name has spaces or special characters?
Slugified the same way knowledge files already are: "Admin Dashboard" → admin-dashboard. The existing slugify() function in tenet-update-knowledge.ts handles this.
What about knowledge files from a previous feature that are relevant to the current one?
Knowledge files are not scoped to a feature — they're project knowledge. knowledge/2026-04-08_oauth-pkce-flow.md is available to all future features. The agent selects relevant knowledge by filename when needed. This is the existing design.
How does the agent know what feature name to use?
During the interview phase, the agent asks the user what they want to build. The answer becomes the feature slug: "I want to add OAuth support" → feature: oauth. This is set once at the start and flows through to all documents and jobs.
What if I want to see all specs across all features?
ls .tenet/spec/ — that's the index. Sorted by date automatically. git log --oneline .tenet/spec/ for commit history across all specs. No generated index file needed.
Does this scale to 50+ features over a year?
50 files in spec/ is perfectly manageable — it's a directory, not a document. ls -la .tenet/spec/ | wc -l tells you how many features you've built. If it gets noisy, ls .tenet/spec/*oauth* filters instantly. Git blame shows when each spec was written and by whom.
What happens to the old singleton files during migration?
If spec/spec.md exists (old format), tenet_compile_context falls back to it. New runs create $date-$feature.md files. The old spec.md stays until manually removed. No breaking change — graceful migration.

Code Changes Required

tenet_compile_context.ts

The biggest change. Currently hardcodes spec/spec.md, spec/decomposition.md.

Before:

const specMd = readIfExists(path.join(tenetPath, 'spec', 'spec.md'));
const decompositionMd = readIfExists(path.join(tenetPath, 'spec', 'decomposition.md'));

After:

// Read feature name from job params
const feature = job.params.feature as string | undefined;

// Resolve latest document by $date-$feature glob pattern
const specMd = feature
  ? resolveLatest(path.join(tenetPath, 'spec'), feature)
  : readIfExists(path.join(tenetPath, 'spec', 'spec.md'));   // fallback for old format

const decompositionMd = feature
  ? resolveLatest(path.join(tenetPath, 'decomposition'), feature)
  : readIfExists(path.join(tenetPath, 'spec', 'decomposition.md'));

const interviewMd = feature
  ? resolveLatest(path.join(tenetPath, 'interview'), feature)
  : '';

resolveLatest(dir, feature):

// Glob: dir/*-{feature}.md
// Sort by filename (date prefix ensures chronological order)
// Return content of last match (latest)
// Return '' if no matches

tenet_register_jobs.ts

Add feature field to job entry schema. This propagates to all jobs in the DAG.

// Input schema adds:
feature: z.string().min(1).describe('Feature slug (e.g. "oauth", "payments")')

// When creating jobs:
params: {
  dag_id: entry.id,
  name: entry.name,
  prompt: entry.prompt,
  depends_on: entry.depends_on,
  feature: input.feature,        // ← NEW: propagated to all jobs
}

Interview / Spec writing (SKILL.md responsibility)

The SKILL.md instructions tell the agent how to write documents during crystallization. These instructions change from "write to spec/spec.md" to "write to spec/$date-$feature.md".

This is a SKILL.md change, not an MCP server code change. The MCP server doesn't write specs or interviews — the agent does.

Decomposition directory: Move from spec/decomposition.md to decomposition/$date-$feature.md. This separates decomposition from spec (they're different artifacts with different purposes).

No changes needed

knowledge/ — already uses $date_$slug.md naming
harness/current.md — project-wide, stays singular
steer/inbox.md — project-wide, stays singular
status/* — auto-generated from DB, always overwritten
tenet_retry_job — feature-agnostic, works on job IDs
tenet_update_knowledge — already date-prefixed
tenet_get_status — reads from DB, not files
state-store.ts — no schema changes needed
job-manager.ts — feature flows through params, no structural change

Summary

What we gain
  • Documents accumulate, never overwrite
  • Full project history in the filesystem
  • Git-trackable evolution of features
  • No index maintenance burden
  • Graceful migration from old format
  • Knowledge stays cross-feature accessible
  • Minimal code changes (compile_context + register_jobs)
What we don't add
  • No index.md (filesystem is the index)
  • No initiative table in SQLite
  • No "active feature" state to manage
  • No session directories
  • No migration scripts
  • No new DB schema
