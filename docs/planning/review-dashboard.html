What is Tenet?

A cross-platform AI agent plugin that enables 12-hour autonomous development cycles with human steering, structured documentation, and recursive task decomposition. Target platforms: Claude Code, OpenAI Codex, OpenCode.

Two Phases

Crystallization (Human-Heavy)
  • Ideation → Interview → Spec + Harness → Dependency Graph
  • Goal: reduce ambiguity until autonomous execution is safe
  • Artifact-anchored ambiguity gate (Clarity ≥ 0.8)
Execution (Agent-Heavy)
  • The 12-hour dev loop through the dependency graph
  • Fresh sessions per job, compiled context bootstrap
  • Human steering as optional async input

Key Principles (Final v0.3)

  1. Markdown files are the management layer — No databases, no proprietary formats
  2. Fresh sessions per job — Primary defense against error compounding and LLM context degradation
  3. Compiled context, not file dumps — Inspired by Google ADK
  4. Confidence-tagged knowledge — Combat memory drift
  5. Purpose alignment over spec compliance — Fill spec holes with intent
  6. Async human steering — Context / Directive / Emergency classification
  7. Leave wrong turns in context — Validated by Manus production experience
  8. Separate generation from validation — Author/Critic split

Design Philosophy

The Error Compounding Problem

The math is unforgiving: 95% per-step accuracy over 20 steps = only 36% end-to-end success. At 100 steps, even 1% error rates → near-certain collapse.

Three Types of Drift

Drift TypeDefinitionMitigation
Reasoning drift Small logical errors compound across turns Fresh sessions per job
Context drift Failed calls, verbose tracebacks crowd out signal Bootstrap compiler filters for recency & relevance
Memory drift Incomplete info at turn 20 becomes ground truth at turn 80 Confidence tags on all knowledge writes

Confidence Tags

## Token refresh strategy [confidence: implemented-and-tested]
Refresh tokens are rotated on each use.

## Rate limiting approach [confidence: implemented-not-tested]
Using token bucket algorithm with 100 req/min per user.

## WebSocket reconnection [confidence: decision-only]
Will use exponential backoff with jitter. Not yet implemented.

Allowed values: decision_only, implemented_not_tested, implemented_and_tested, observed_from_existing_code, superseded

Scale-Adaptive Execution Modes NEW

Not all work requires full crystallization. Execution overhead scales with task complexity:

ModeUse CasePipelineQuality Gate
Full Mode New features, major refactors, greenfield Full interview → spec → decomposition → execution Ambiguity ≥ 0.8
Standard Mode Well-understood features, moderate complexity Brief clarification → quick spec → execution (±1 job) Harness + 2-3 scenarios
Quick Mode Bug fixes, typos, config changes Direct execution with harness enforcement; skip interview/spec/decomposition Mechanical eval (lint/test)

Core principle: Harness quality enforcement (lint, format, test) applies at ALL modes. Only planning overhead scales down. Mechanical eval is non-negotiable.

Phase 1: Crystallization (Interview)

Artifact-Anchored Ambiguity Gate

The interview ends when enough artifacts exist, not when the user "feels done":

DimensionWeightScore 1.0Score 0.5Score 0.0
Goal clarity0.4Written acceptance test existsQualitative description ("feels fast")No goal stated
Constraints0.3Harness constraints + danger zonesImplicit ("use existing stack")No constraints discussed
Success criteria0.3Measurable (<1s load time)Qualitative ("easy to use")"I'll know it when I see it"

Gate: Clarity ≥ 0.8 (Ambiguity ≤ 0.2). Scoring is deterministic — artifact exists or doesn't.

Interview Outputs

  • Transcript with decisions and rationale
  • Rejected alternatives (why NOT)
  • Ambiguity score history with artifact evidence
  • Scenario-based acceptance criteria (concrete user stories)
  • Anti-scenarios (what "bad" looks like)
  • Visual artifacts: self-contained HTML files (embedded SVG/CSS/JS, no external dependencies) in .tenet/visuals/ for UI mockups, architecture diagrams, flow diagrams, and data model diagrams

Phase 2: Spec + Harness

Spec (WHAT to build)
  • Acceptance criteria per component
  • Scenario-based ACs
  • Measurable targets
  • Reference artifacts (self-contained HTML visualizations in .tenet/visuals/)
  • Property specifications (for property-based testing)
Harness (HOW to operate)
  • Danger zones (files/dirs never to modify)
  • Iron laws (invariants)
  • Code formatting & linting rules (strictly enforced)
  • Unit test coverage requirements
  • E2E testing requirements
  • Architecture rules (no circular deps, validation middleware, etc.)
  • Code style principles (composition over inheritance, etc.)
  • Testing strategy (what must be tested, how)

Phase 3: Dependency Graph

Complexity-Based Job Sizing

## Job decomposition criteria
- Logical cohesion: one coherent concern per job
- Context requirements: manageable context window load
- Interface boundaries: natural split at well-defined APIs
- Quality gates: testable acceptance criteria per job

Jobs are decomposed based on logical cohesion, context requirements, and natural interface boundaries — keeping them small enough to avoid LLM quality degradation at long context lengths.

Bootstrap Compiler

New in v0.3: Instead of loading raw files, a compilation step produces a minimal Working Context per job. Inspired by Google ADK's "compiled context" architecture.

Pipeline

Raw state (all .tenet/ files)
  → Relevance filter: which files matter for THIS job?
  → Recency filter: current vs. historical within each file?
  → Interface extraction: only interfaces this job touches
  → Confidence filter: prioritize [implemented-and-tested] over [decision-only]
  → Steer integration: fold in pending steer messages for this job
  → Todo recitation: append job objectives at END of context
  → Working Context: single compiled document

Why Todo Recitation Matters

Manus discovered that rewriting a todo list pushes the global plan into the model's recent attention span. This exploits the U-shaped attention curve (models attend well to beginning and end, poorly to middle) to keep current job goals in high-attention positions.

KV-Cache Optimization

KV-cache optimization reduces repeated prompt recomputation and improves long-run execution efficiency across sessions.

  • Stable prompt prefixes: Harness + system instructions occupy the same prefix across sessions
  • Append-only contexts: Never rewrite earlier context within a session
  • Cache breakpoints: Stable portion (harness, spec) first, then job-specific portion

In-Session Checkpointing

ThresholdAction
Quality degradation signalsAgent should checkpoint (repeated failures, going in circles, re-reading same files)
Stagnation detectedSession restarts with progress snapshot + job spec
Repeated file readsImmediate checkpoint (detects "Endless File Reading" pattern)

Error Preservation

Key insight (Manus): Keep failed actions and stack traces in the working context within a session. "Erasing failure removes evidence." Move to lessons.md only at job completion.

Catastrophic Recovery NEW

System resilience principle: Every failure state must be recoverable to the last committed checkpoint via git. The system is never corrupted beyond recovery.

Failure TypeRoot CauseRecovery Action
.tenet/ Corruption Partial write, merge conflict, or agent malfunction Restore from git (`.tenet/` should be committed at every checkpoint)
Harness Violation Agent ignores danger zone or iron laws Immediate halt + EMERGENCY alert + wait for user + git revert to last checkpoint
Broken Build State Code changes leave project non-compilable Git stash to last passing checkpoint, retry or escalate with evidence

Invariant: .tenet/ directory is part of version control. On session end, always `git add .tenet/` before proceeding. This ensures state snapshots are audit-trail compatible.

Evaluation Pipeline

Core principle: Never trust AI to both generate and validate. Generate and verify must be separate steps, ideally with different context.

5-Stage Pipeline

StageWhatKey Innovation
1: Mechanical Lint, build, type-check, test suite Catches 60-80% of issues
1.5: Property-based NEW Auto-generated fuzz/property tests Properties specified during SPEC phase, before implementation
2: Spec compliance Check ACs, scope creep/reduction, doc-code sync Doc-code sync check
3: Purpose alignment NEW Critic session with adversarial framing Critic gets spec + diff only, NOT author's reasoning; zero findings are suspicious and must be re-justified
4: Self-questioning Structured categories: edge cases, security, performance... 3-5 questions per relevant category

Author/Critic Split Details

Author Session Receives
  • Full compiled context
  • Knowledge docs
  • Previous attempt failures
  • Steer messages
Critic Session Receives
  • Spec / acceptance criteria / scenarios
  • Harness constraints
  • Code diff only
  • Anti-scenarios
  • Zero-findings rule: if critic returns 0 findings, it must re-analyze from a different perspective or explicitly justify why zero is legitimate
  • NOT: author's reasoning, tests, or knowledge docs
Critic gate hardening: Zero findings is treated as suspicious, not automatically clean. The critic must either produce a second-pass analysis using a different framing (e.g., edge-case, security, integration, purpose) or provide explicit evidence-backed justification for a legitimate zero-findings outcome.

Three Types of Cascade Checks

Improvement over v0.1: The original PRD bundled all cascade operations into one vague "cascade check." The review correctly identified these as fundamentally different operations.

Type 1: Document-to-Document Alignment

›
  • Input: Two markdown docs (e.g., interview ↔ spec)
  • Question: "Do these documents contradict each other?"
  • Trigger: Any upstream document update
  • Examples: interview vs spec, spec vs harness, scenario vs anti-scenario

Type 2: Code-to-Document Alignment

›
  • Input: Knowledge doc + the actual code it describes
  • Question: "Does this document accurately describe what the code does?"
  • Trigger: After every completed job
  • Critical: Knowledge docs MUST reflect implementation reality

Type 3: Trajectory-to-Purpose Alignment (Drift Detection)

›
  • Input: Interview artifacts + status + recent knowledge changes
  • Question: "Is the project still heading toward the original goal?"
  • Trigger: Every N iterations (default 3), or after major steer directive

Async Human Steering

Mechanism

User writes to steer/inbox.md at any time. Agent checks at natural checkpoints (after eval, before next job). Check is lightweight: compare file mtime without recompiling context.

Message Classification

ClassPrefixActionTiming
Context (none — default) Folded into next job's compiled context No disruption
Directive DIRECTIVE: Updates queue ordering, may reopen spec alignment After current job
Emergency EMERGENCY: Halt immediately, create blocking incident job Immediate

Multi-Job Tagging

## 2026-04-02 14:30 [affects: auth-oauth, auth-basic, api-endpoints]
OAuth should use PKCE flow, not implicit grant

Orchestrator auto-tags steer messages with affected job IDs. Untagged messages go to all pending jobs.

Message Acknowledgment Tracking

Each message in steer/inbox.md carries inline lifecycle status so the user can see processing state immediately: received → acknowledged → acted_on → resolved. The agent writes status transitions and responses directly in the same file.

## 2026-04-06 10:12 [affects: api-auth] [status: acknowledged]
DIRECTIVE: Enforce PKCE for OAuth public clients.

Agent response (2026-04-06 10:14): Acknowledged. Reordered queue; adding auth hardening job.

## 2026-04-06 11:05 [affects: api-auth] [status: acted_on]
Agent response: Patch implemented and under evaluation.

## 2026-04-06 11:22 [affects: api-auth] [status: resolved]
Agent response: Merged with tests passing and docs updated.
Evolution from v0.1: Original had 4 types (course correction, spec change, pause/redirect, new feature) with URGENT prefix. v0.3 simplifies to 3 classes (context/directive/emergency) based on the insight that steer should usually be context, not commands.

Concern-Based Organization

Knowledge docs are organized by concern (auth, data model, API), NOT by phase (requirements, architecture). When the agent implements auth, it reads knowledge/auth.md and writes back to the same file.

Tiered Compaction

current.md   → Always up to date. Agents read THIS.
changelog.md → What changed and why. Append-only.
archive/     → Summarized older changelogs.

Trigger: When changelog.md exceeds 50 entries or ~10K tokens.

Document Health Mechanisms

Index + Periodic Health Audit

›
  • .tenet/index.md: Auto-generated table of contents with file inventory, last-updated timestamps, short summaries, and status indicators.
  • Orphan detection: Flags files not referenced by any active document chain.
  • Staleness checks: Detects knowledge docs drifting out of sync with implementation.
  • Missing update checks: Detects completed jobs without required documentation updates.
  • Reference integrity: Catches broken cross-document references and invalid links.
  • Steer responsiveness: Alerts on unacknowledged messages in steer/inbox.md aging beyond threshold.

Open Questions

1. Ambiguity Gate Calibration

›

The 0.2 threshold may need tuning per project type (greenfield vs brownfield, frontend vs backend). Artifact-anchored approach reduces but doesn't eliminate subjectivity in level descriptors.

2. Knowledge Doc Granularity

›

Rule of thumb: one file per bounded context (DDD sense). If a knowledge file exceeds 5K tokens, consider splitting by sub-concern.

3. Parallel Job Execution

›

Current design assumes sequential. Cursor's experience: self-coordinating agents with shared files fail — need Planner/Worker hierarchy. Phase 6+ complexity.

4. Model Selection Per Phase

›

Interview and purpose alignment may need frontier models. Mechanical dev may work with lighter-weight models. Cached inference economics can differ significantly from uncached paths.

5. Critic Model Selection

›

Author/critic split works best with different models or at minimum different temperature/sampling. Same model with different context is better than nothing but may share blind spots.

6. Property Specification Completeness

›

Properties must be specified before implementation, but users may not know all upfront. Allow iterative surfacing during interview; let harness accumulate them.

7. Team Collaboration

›

Multiple users steering simultaneously or multiple agents on same project introduces coordination challenges beyond current single-user design.
